# Requirements Document

## Introduction

This feature integrates ESLint and Prettier into the existing GitHub Actions monorepo to enforce consistent code quality and formatting standards across all workspaces. ESLint provides TypeScript-aware linting with JSDoc enforcement, Prettier handles code formatting, and both are integrated into the existing CI workflow alongside npm audit security checks and coverage report artifacts.

## Glossary

- **Monorepo**: The root repository at `/Volumes/Projects/GitHub/LDK-Systems/actions/` containing multiple npm workspaces
- **Workspace**: An individual npm package within the monorepo (extract-dotnet-version, check-release-version, generate-release-notes, packages/lib)
- **Linter**: ESLint, a static analysis tool that identifies problematic patterns in TypeScript/JavaScript code
- **Formatter**: Prettier, an opinionated code formatter that enforces consistent style
- **CI_Workflow**: The GitHub Actions workflow defined in `.github/workflows/ci.yml` that runs on pull requests
- **Root_Config**: Configuration files placed at the monorepo root that apply to all workspaces
- **Flat_Config**: ESLint's modern configuration format using `eslint.config.mjs` files
- **JSDoc**: Documentation comments in source code that describe function signatures, parameters, return types, and purpose
- **Coverage_Report**: Test coverage output generated by Vitest showing which lines and branches are exercised by tests

## Requirements

### Requirement 1: ESLint Configuration

**User Story:** As a developer, I want ESLint configured at the monorepo root with TypeScript-aware rules, so that all workspaces share consistent linting standards without duplicated configuration.

#### Acceptance Criteria

1. THE Root_Config SHALL include an ESLint flat configuration file (`eslint.config.mjs`) at the monorepo root
2. THE Linter SHALL use `typescript-eslint` with typed linting enabled, configured to reference each workspace's `tsconfig.json` for type-aware rule analysis
3. THE Linter SHALL apply the `recommended` named rule sets from both ESLint core (`eslint.configs.recommended`) and typescript-eslint (`tseslint.configs.recommended`)
4. THE Linter SHALL target all `.ts` files within `src/` and `__tests__/` directories of each workspace defined in the root `package.json` workspaces field
5. THE Linter SHALL exclude `dist/`, `node_modules/`, and `coverage/` directories from analysis via the configuration's ignore patterns
6. WHEN a developer runs `npm run lint` from the monorepo root, THE Linter SHALL analyze all workspace source files in a single invocation and exit with code 0 if no violations are found

### Requirement 2: Prettier Configuration

**User Story:** As a developer, I want Prettier configured at the monorepo root, so that all code follows a consistent formatting style without manual intervention.

#### Acceptance Criteria

1. THE Root_Config SHALL include a Prettier configuration file (`.prettierrc`) at the monorepo root that defines at minimum the following options: print width, tab width, semicolon usage, quote style, and trailing comma behavior
2. THE Formatter SHALL enforce formatting for all `.ts` files within `src/` and `__tests__/` directories across workspaces
3. THE Root_Config SHALL include a `.prettierignore` file that excludes `dist/`, `node_modules/`, and `package-lock.json` from formatting
4. WHEN a developer runs the format check command from the monorepo root, THE Formatter SHALL report each file that does not conform to the configured style and exit with a non-zero exit code if one or more non-conforming files are found
5. WHEN a developer runs the format write command from the monorepo root, THE Formatter SHALL automatically rewrite all non-conforming files to match the configured style

### Requirement 3: ESLint and Prettier Compatibility

**User Story:** As a developer, I want ESLint and Prettier to work together without conflicts, so that formatting rules do not trigger linting errors.

#### Acceptance Criteria

1. THE Linter SHALL disable all formatting-related rules that conflict with Prettier by extending `eslint-config-prettier` as the last configuration in the flat config array
2. WHEN a file passes Prettier formatting checks, THE Linter SHALL report zero violations originating from rules disabled by `eslint-config-prettier` for that file
3. WHEN a developer runs the lint command and the format check command sequentially on the same file set, THE Linter and Formatter SHALL not produce contradictory guidance where fixing one tool's violation would introduce a violation in the other

### Requirement 4: JSDoc Enforcement

**User Story:** As a developer, I want JSDoc comments required on all exported functions and classes, so that the codebase is well-documented and easier to understand.

#### Acceptance Criteria

1. THE Linter SHALL enforce the presence of JSDoc comments on all exported functions, classes, and interfaces in `src/` directories
2. WHEN a JSDoc comment exists on an exported function, THE Linter SHALL require `@param` tags for each named parameter of that function
3. THE Linter SHALL require JSDoc `@returns` tags for all exported functions whose return type is not `void`
4. WHEN a JSDoc comment block exists, THE Linter SHALL require a non-empty description of at least one word in that block
5. WHEN an exported function, class, or interface in a `src/` directory lacks a JSDoc comment, THE Linter SHALL report a violation
6. THE Linter SHALL not require JSDoc comments on files within `__tests__/` directories
7. THE Linter SHALL not require JSDoc comments on exported type aliases

### Requirement 5: NPM Scripts

**User Story:** As a developer, I want convenient npm scripts to run linting and formatting checks, so that I can quickly validate code locally before pushing.

#### Acceptance Criteria

1. THE Monorepo root `package.json` SHALL include a `lint` script that runs ESLint across all workspaces
2. THE Monorepo root `package.json` SHALL include a `format:check` script that runs Prettier in check mode across all workspaces
3. THE Monorepo root `package.json` SHALL include a `format` script that runs Prettier in write mode to auto-fix formatting issues
4. THE Monorepo root `package.json` SHALL include a `lint:fix` script that runs ESLint with auto-fix enabled across all workspaces
5. WHEN a developer runs `npm run lint` and no violations are found, THE Linter SHALL exit with code 0
6. IF violations are found in any workspace file when a developer runs `npm run lint`, THEN THE Linter SHALL exit with a non-zero code regardless of git status
7. WHEN a developer runs `npm run format:check` and all files conform to the configured Prettier style, THE Formatter SHALL exit with code 0
8. IF any files do not conform to the configured Prettier style when a developer runs `npm run format:check`, THEN THE Formatter SHALL exit with a non-zero code
9. IF unfixable violations remain in the current run after a developer runs `npm run lint:fix`, THEN THE Linter SHALL exit with a non-zero code

### Requirement 6: CI Workflow Integration — Lint and Format

**User Story:** As a team lead, I want lint and format checks to run automatically on pull requests, so that code quality standards are enforced before merging.

#### Acceptance Criteria

1. THE CI_Workflow SHALL include a quality job that runs the `npm run lint` script to execute ESLint on all workspace `.ts` files within `src/` and `__tests__/` directories
2. THE CI_Workflow SHALL include a format check step within the quality job that runs the `npm run format:check` script to verify all workspace TypeScript files conform to Prettier standards
3. WHEN the lint step returns a non-zero exit code (whether from code violations or tool errors), THE CI_Workflow SHALL fail the quality job and report the failure on the pull request check
4. WHEN the format check step detects non-conforming files, THE CI_Workflow SHALL fail the quality job and report a non-zero exit code on the pull request check
5. THE CI_Workflow SHALL enforce that the existing test and build jobs depend on the successful completion of the quality job via the `needs` configuration, so that lint and format failures prevent test and build execution

### Requirement 7: CI Workflow Integration — Security Audit

**User Story:** As a team lead, I want npm dependency security audits to run on pull requests, so that known vulnerabilities are caught before merging.

#### Acceptance Criteria

1. THE CI_Workflow SHALL include a security audit step that runs `npm audit` against all workspace dependencies (production and development) at the repository root level
2. WHEN `npm audit` returns a non-zero exit code for any reason (including moderate+ vulnerabilities, registry connectivity errors, or malformed package files), THE CI_Workflow SHALL fail the pull request check
3. THE CI_Workflow SHALL run the security audit step within the lint/quality job, before the build job, so that vulnerability feedback is available without waiting for compilation or tests
4. THE CI_Workflow SHALL fail the security audit for any moderate or higher vulnerabilities regardless of whether they are in production or development dependencies

### Requirement 8: CI Workflow Integration — Coverage Artifacts

**User Story:** As a team lead, I want test coverage reports uploaded as CI artifacts, so that the team can review coverage trends and identify untested code.

#### Acceptance Criteria

1. WHEN the test job executes, THE CI_Workflow SHALL generate a coverage report using Vitest's v8 coverage provider in both JSON summary and HTML formats
2. WHEN the test job completes, THE CI_Workflow SHALL upload the coverage report directory as a GitHub Actions artifact named "coverage-report"
3. THE Coverage_Report artifact SHALL be retained for a minimum of 30 days
4. THE Coverage_Report SHALL include summary metrics for lines, branches, functions, and statements, as well as per-file breakdowns of the same metrics
5. IF the test job fails, THEN THE CI_Workflow SHALL still upload the coverage report artifact for any tests that did execute

### Requirement 9: Developer Dependencies

**User Story:** As a developer, I want all linting and formatting dependencies installed as devDependencies at the monorepo root, so that workspace packages remain lightweight.

#### Acceptance Criteria

1. THE Monorepo root `package.json` SHALL list ESLint, Prettier, typescript-eslint, eslint-config-prettier, eslint-plugin-jsdoc, and any additional ESLint plugins referenced by the Root_Config as `devDependencies`
2. THE Root_Config SHALL not require any workspace-level ESLint or Prettier configuration files, and no workspace `package.json` SHALL list ESLint, Prettier, or their plugins as dependencies or devDependencies; workspace-level ESLint or Prettier dependencies SHALL be strictly prevented
3. WHEN a developer runs `npm ci` from the monorepo root, THE Monorepo SHALL install all linting and formatting tools such that `npm run lint` and `npm run format:check` are executable without additional manual steps
